## 三、面试官的追问逻辑

面试官通常会用一个问题作为入口，然后层层深入：

**text**

```
Q: 你们怎么用 Redis 的？
A: 做缓存，热点数据存 Redis。
Q: 热点数据怎么定义的？缓存击穿怎么处理的？
A: 用 singleflight 合并请求。
Q: singleflight 的原理是什么？有什么坑？
A: 共享请求结果，但要注意超时和 panic 处理。
Q: 如果 Redis 挂了怎么办？
A: 本地缓存兜底 + 熔断 + 报警。
Q: 本地缓存怎么保证一致性？
A: 订阅 Redis 过期事件，但会有延迟，业务可接受最终一致。
```

这种追问方式考察的是： **你只是知道名词，还是真的深入理解并实践过** 。

# 熟练 K8s ，要掌握的大厂面试题

大厂对于“熟练 Kubernetes”的要求，远不止会使用 `kubectl` 命令。面试官期待的是你能 **深入理解 K8s 的底层原理、核心组件的实现细节，并具备基于 K8s 生态解决实际问题的能力** 。下面我从**基础原理、调度与扩展、网络与存储、排障与优化、生态与扩展**五个维度，为你梳理高频面试题和加分回答方向。

---

## 一、基础原理（必考）

### 1. 一个 Pod 从创建到运行的完整过程

 **考察点** ：对 K8s 核心组件交互的理解。

 **加分回答** ：

**text**

```
1. kubectl 提交 YAML → API Server（认证/鉴权/准入控制）
2. API Server 将 Pod 对象写入 etcd
3. Scheduler 通过 Watch 发现未调度的 Pod，经过预选/优选，选出 Node
4. Scheduler 将绑定结果写回 API Server
5. kubelet 通过 Watch 发现分配到本节点的 Pod，调用 CRI 创建容器
6. kubelet 上报 Pod 状态，API Server 更新 etcd
7. Controller Manager 持续调谐，确保副本数符合期望
```

 **深入追问** ：如果 Scheduler 挂了怎么办？Pod 还能创建吗？

* 已调度的 Pod 不受影响，新 Pod 会 Pending，直到 Scheduler 恢复。

### 2. etcd 在 K8s 中的作用？为什么是 K8s 的瓶颈？

 **考察点** ：对控制平面核心的理解。

 **加分回答** ：

* **作用** ：存储所有集群状态数据（Pod、Service、ConfigMap 等）
* **瓶颈原因** ：
* 磁盘 IOPS 敏感（建议 SSD）
* 网络延迟敏感
* 单集群默认存储上限约 8GB
* **优化方向** ：硬件（SSD+万兆网络）、定期压缩历史版本、控制集群规模（单集群 < 5000 节点）

---

## 二、调度与扩展（高频）

### 3. Scheduler 的调度流程？如何实现自定义调度？

 **考察点** ：调度器扩展能力。

 **加分回答** ：

* **流程** ：预选（过滤不满足条件的 Node）→ 优选（打分排序）→ 绑定
* **自定义调度** ：
* **方式一** ：扩展调度器插件（实现 Filter/Score 接口）
* **方式二** ：独立调度器，监听 `spec.schedulerName` 为自定义名称的 Pod

### 4. 如何实现一个 Operator？Controller 的核心机制是什么？

 **考察点** ：K8s 扩展机制（controller-runtime）。

 **加分回答** ：

* **核心机制** ： **控制循环** （Reconcile Loop）
* Informer：通过 List+Watch 从 API Server 同步资源
* WorkQueue：去重、限速、重试
* Reconcile 函数：对比期望状态与实际状态，执行调谐操作
* **实现** ：使用 `controller-runtime` 框架，定义 CRD，实现 `Reconcile` 方法

### 5. Pod 优先级与抢占（Priority & Preemption）是如何工作的？

 **考察点** ：调度器深度理解。

 **加分回答** ：

* 高优先级 Pod 调度失败时，可以抢占低优先级 Pod
* 抢占过程：Scheduler 寻找可驱逐的 Pod 集合，删除这些 Pod 释放资源
* **关键点** ：`PriorityClass` 定义优先级值，抢占可能造成服务抖动

---

## 三、网络与存储（核心）

### 6. Service 的 ClusterIP 是如何实现的？kube-proxy 的 iptables 和 IPVS 模式区别？

 **考察点** ：网络模型理解。

 **加分回答** ：

* **实现** ：kube-proxy 在每台 Node 上维护转发规则
* **iptables 模式** ：随机选择后端，规则链式匹配，量大时性能下降
* **IPVS 模式** ：基于内核哈希表，O(1) 查找，支持多种负载均衡算法（rr、lc、sh 等）
* **选择** ：大规模集群（> 1000 Service）建议用 IPVS

### 7. CNI 的作用？常见的 CNI 插件有哪些？

 **考察点** ：网络插件理解。

 **加分回答** ：

* **作用** ：容器网络接口，负责 Pod 网络配置（IP 分配、网络连通性）
* **常见插件** ：
* Calico：支持网络策略，性能较好
* Flannel：简单轻量，适合小集群
* Cilium：基于 eBPF，支持可观测性、安全策略
* **选型** ：需要网络策略 → Calico/Cilium；只需基础连通 → Flannel

### 8. PV、PVC、StorageClass 的关系？动态供给流程？

 **考察点** ：存储体系理解。

 **加分回答** ：

* **PV** ：集群存储资源
* **PVC** ：用户存储请求
* **StorageClass** ：动态供给模板，指定 provisioner（如云盘插件）
* **动态供给流程** ：

1. 用户创建 PVC（指定 StorageClass）
2. 该 StorageClass 的 Provisioner 自动创建 PV
3. PVC 与 PV 自动绑定

---

## 四、排障与优化（工程经验）

### 9. Pod 一直 Pending，如何排查？

 **考察点** ：问题定位思路。

 **加分回答** ：

**text**

```
1. kubectl describe pod → 看 Events（资源不足？PVC 未绑定？节点污点？）
2. 资源不足：查看节点资源使用情况
3. 节点污点：kubectl describe node，检查 Taints
4. PVC 未绑定：查看 PV/PVC 状态
5. 调度器问题：查看 kube-scheduler 日志
```

### 10. 节点 NotReady，可能原因及排查步骤？

 **考察点** ：节点故障处理。

 **加分回答** ：

* **可能原因** ：kubelet 停止、磁盘空间满、网络不通、内存/CPU 压力
* **排查步骤** ：

1. 查看节点状态：`kubectl describe node`
2. 登录节点，检查 kubelet 状态：`systemctl status kubelet`
3. 查看 kubelet 日志：`journalctl -u kubelet -f`
4. 检查磁盘：`df -h`
5. 检查网络：`ping` 其他节点

### 11. 如何对 K8s 集群做性能优化？

 **考察点** ：生产环境经验。

 **加分回答** ：

* **控制平面** ：etcd 使用 SSD、API Server 水平扩展、调整 `--max-requests-inflight`
* **节点** ：配置 `--system-reserved` 预留系统资源，使用 `topology-manager` 优化 NUMA 绑定
* **调度** ：调整调度器配置（如 `percentage-of-nodes-to-score`）
* **网络** ：使用 IPVS 模式，选择合适的 CNI
* **监控** ：Prometheus 监控集群指标，设置告警阈值

## 五、生态与扩展（加分项）

### 12. Helm 和 Kustomize 的区别？分别适用于什么场景？

 **考察点** ：包管理与配置管理理解。

 **加分回答** ：

* **Helm** ：模板引擎 + 包管理，适合分发复杂应用（如 Prometheus、MySQL）
* **Kustomize** ：原生 K8s 配置覆盖，无需模板，适合多环境管理（dev/staging/prod）
* **选择** ：对外分发 → Helm；内部多环境 → Kustomize

### 13. Service Mesh（如 Istio）解决了什么问题？有哪些核心组件？

 **考察点** ：云原生网络治理理解。

 **加分回答** ：

* **解决的问题** ：将服务治理（熔断、重试、金丝雀发布）从业务代码下沉到基础设施
* **核心组件** ：
* Envoy：Sidecar 代理
* Pilot：配置管理
* Mixer：策略控制（已逐渐被 Envoy 插件替代）
* Citadel：安全证书管理
* **优缺点** ：多语言兼容，但增加延迟（约 5-10ms），资源开销

### 14. 如何实现多集群管理？

 **考察点** ：大规模集群治理。

 **加分回答** ：

* **方式一** ：联邦（Federation），统一 API 入口
* **方式二** ：多云管理平台（如 Rancher），集中管理多个集群
* **方式三** ：集群注册 + 统一 GitOps（如 ArgoCD 多集群部署）
* **典型场景** ：开发/测试/生产隔离、跨云容灾

# 大厂高并发考点面试题

大厂高并发面试的核心，是考察你在**高流量、高可用、高扩展**场景下的**架构设计能力**和 **工程落地经验** 。面试官不会只问“什么是缓存穿透”，而是会追问：“如果让你设计一个秒杀系统，QPS 10万+，你怎么做？瓶颈在哪里？如何保证数据一致性？”

下面从**核心理论基础、缓存、数据库、消息队列、微服务治理、实战案例**六个维度，梳理高频考点和回答要点。

---

## 一、核心理论基础（必考）

### 1. 横向扩展 vs 纵向扩展

 **问题** ：什么时候用横向扩展，什么时候用纵向扩展？

 **加分回答** ：

* **纵向扩展** ：升级单机硬件（CPU、内存、磁盘），适用于状态复杂、难以拆分的系统
* **横向扩展** ：增加机器数量，适用于无状态服务
* **大厂选择** ：优先横向扩展，因为单机有物理上限，且云原生时代机器是“牲口”不是“宠物”

### 2. 可用性指标（SLA）

 **问题** ：99.99% 和 99.9% 的可用性差别有多大？

 **加分回答** ：

* 99.9%：一年允许 downtime 约 8.76 小时
* 99.99%：一年允许 downtime 约 52.6 分钟
* 99.999%：一年允许 downtime 约 5.26 分钟
* **设计目标** ：根据业务定 SLA，核心链路追求 99.99%，非核心可适当降低

---

## 二、缓存（最常考）

### 3. 缓存穿透、击穿、雪崩的区别及解决方案

| 问题           | 描述                               | 解决方案                         |
| -------------- | ---------------------------------- | -------------------------------- |
| **穿透** | 查询不存在的数据，请求直达 DB      | 布隆过滤器、缓存空对象（短时）   |
| **击穿** | 热点 key 过期，大量请求同时打到 DB | 互斥锁（singleflight）、逻辑过期 |
| **雪崩** | 大量 key 同时过期，DB 压力骤增     | 过期时间加随机值、多级缓存       |

 **加分回答** ：

* **singleflight** ：合并同一 key 的并发请求，只有一个请求去 DB 查，其他等待结果
* **布隆过滤器** ：用 bit 数组快速判断 key 是否存在，有误判率但可控
* **实际经验** ：“我们曾用 singleflight + 本地缓存，将击穿时的 DB QPS 从 5万 降到 500”

### 4. 缓存和数据库的一致性如何保证？

 **加分回答** ：

* **策略** ：先更新 DB，后删除缓存（而不是更新缓存）
* **为什么** ：更新缓存可能引入并发脏数据，删除缓存让下次读重建
* **兜底** ：引入消息队列重试删除，订阅 binlog 异步删除
* **最终一致性** ：允许短暂不一致，业务可接受

### 5. Redis 的持久化机制？如何选型？

 **加分回答** ：

* **RDB** ：快照，适合备份，恢复快，但可能丢数据
* **AOF** ：日志，数据更安全，但文件大、恢复慢
* **混合持久化（Redis 4.0+）** ：RDB + AOF，兼顾恢复速度和数据安全
* **选型** ：缓存场景用 RDB 或关闭；核心数据用混合持久化

---

## 三、数据库（分库分表）

### 6. 分库分表的时机和策略

 **加分回答** ：

* **时机** ：单表数据 > 500 万行，或单表大小 > 10GB，或 TPS > 2000
* **分表策略** ：
* 范围分片：按时间、ID 范围，扩容简单，但热点不均
* 哈希分片：按 user_id % N，数据均匀，但扩容需迁移
* 一致性哈希：扩容只影响相邻节点，适合动态扩缩容
* **分库分表后的挑战** ：跨分片 JOIN、分布式事务、分页查询

### 7. 分布式事务解决方案

 **加分回答** ：

* **2PC（两阶段提交）** ：强一致，性能差，适合银行等场景
* **TCC（Try-Confirm-Cancel）** ：业务层实现，性能好，开发复杂
* **本地消息表 + 消息队列** ：最终一致，适合大多数业务
* **Saga** ：长事务补偿，适合跨多个服务

---

## 四、消息队列（削峰填谷）

### 8. 消息队列在秒杀系统中的作用

 **加分回答** ：

* **削峰** ：将瞬时流量平缓到更长的时间窗口，保护下游系统
* **异步** ：下单后立即返回“处理中”，实际扣库存异步完成
* **解耦** ：订单创建、库存扣减、积分发放等模块解耦
* **流量控制** ：设置消费速度上限，防止 DB 过载

### 9. Kafka 和 RocketMQ 的选型

| 维度     | Kafka              | RocketMQ           |
| -------- | ------------------ | ------------------ |
| 吞吐     | 极高（百万级/秒）  | 高（十万级/秒）    |
| 延迟     | 毫秒级             | 毫秒级             |
| 事务消息 | 不支持             | 支持               |
| 顺序消息 | 分区内有序         | 支持全局/分区有序  |
| 适用场景 | 日志采集、大数据流 | 交易消息、金融场景 |

---

## 五、微服务治理

### 10. 限流算法及选型

 **加分回答** ：

* **令牌桶** ：允许突发，适合 API 网关
* **漏桶** ：恒定速率，适合保护下游 DB
* **滑动窗口** ：精确控制，适合防刷
* **分布式限流** ：Redis + Lua，或本地预分配配额

### 11. 熔断与降级的区别

 **加分回答** ：

* **熔断** ：下游故障时快速失败，防止雪崩。三态：关闭 → 开启 → 半开
* **降级** ：业务高峰期，牺牲非核心功能保核心。如返回兜底数据、关闭推荐服务

### 二、技术决策选型

#### 决策1：Python → Go 全量重构

 **背景** ：zData 数据库一体机原使用 Python 开发，面临性能瓶颈和并发能力不足。

| 决策维度           | Python（旧）        | Go（新）             | 决策依据                 |

| **并发模型** | GIL限制，多进程复杂 | 协程模型，轻量级并发 | IO密集型场景，Go天然优势 |

```
type RingBuffer struct {
    data   []*DataPoint
    head   int  // 写指针
    tail   int  // 读指针
    size   int
}

func (rb *RingBuffer) Write(point *DataPoint) bool {
    if (rb.head+1)%rb.capacity == rb.tail {
        return false  // 缓冲区满
    }
    rb.data[rb.head] = point
    rb.head = (rb.head + 1) % rb.capacity
    return true
}

func (rb *RingBuffer) Read() *DataPoint {
    if rb.tail == rb.head {
        return nil  // 缓冲区空
    }
    point := rb.data[rb.tail]
    rb.tail = (rb.tail + 1) % rb.capacity
    return point
}
```

 **算法价值** ：无锁设计，避免内存分配，适合高频写入场景。

### 二、算法应用场景

#### 1. 限流算法（令牌桶）—— 防止突发流量打垮系统

 **业务场景** ：zData X 存储节点，限制并发请求数，保护后端。

**go**

```
// 令牌桶限流器
type TokenBucket struct {
    capacity   int           // 桶容量
    tokens     int           // 当前令牌数
    rate       int           // 每秒补充令牌数
    lastTime   time.Time
    mu         sync.Mutex
}

func (tb *TokenBucket) Allow() bool {
    tb.mu.Lock()
    defer tb.mu.Unlock()

    // 补充令牌
    now := time.Now()
    elapsed := now.Sub(tb.lastTime)
    tb.tokens += int(elapsed.Seconds() * float64(tb.rate))
    if tb.tokens > tb.capacity {
        tb.tokens = tb.capacity
    }
    tb.lastTime = now

    if tb.tokens > 0 {
        tb.tokens--
        return true
    }
    return false
}
```

 **算法价值** ：平滑流量，允许短时突发，保护系统稳定性。

#### 2. 滑动窗口（Sliding Window）—— 限流与指标统计

 **业务场景** ：API 限流、统计 QPS/TP99。

**go**

```
// 滑动窗口统计 QPS
type SlidingWindow struct {
    buckets   []int   // 时间桶数组
    bucketSize int    // 每个桶的时间窗口（毫秒）
    current   int     // 当前桶索引
    lastTime  time.Time
}

func (sw *SlidingWindow) Increment() {
    now := time.Now()
    elapsed := now.Sub(sw.lastTime).Milliseconds()

    // 滑动窗口
    if elapsed > int64(sw.bucketSize) {
        slots := int(elapsed / int64(sw.bucketSize))
        for i := 0; i < slots && i < len(sw.buckets); i++ {
            sw.current = (sw.current + 1) % len(sw.buckets)
            sw.buckets[sw.current] = 0
        }
        sw.lastTime = now
    }

    sw.buckets[sw.current]++
}

func (sw *SlidingWindow) QPS() int {
    total := 0
    for i := 0; i < len(sw.buckets); i++ {
        total += sw.buckets[i]
    }
    return total * 1000 / (len(sw.buckets) * sw.bucketSize)
}
```

 **算法价值** ：精确统计短时流量，支持突发检测。

#### 3. 一致性哈希（Consistent Hash）—— 设备路由与数据分片

 **业务场景** ：IoT 平台设备接入，将设备哈希到不同的消息队列消费者。

**go**

```
// 一致性哈希环
type ConsistentHash struct {
    ring       map[uint64]string  // 哈希值 -> 节点
    sortedKeys []uint64           // 排序的哈希值列表
    replicas   int                // 虚拟节点数
}

func (ch *ConsistentHash) Get(key string) string {
    if len(ch.ring) == 0 {
        return ""
    }
    hash := ch.hash(key)
    // 二分查找最近的节点
    idx := sort.Search(len(ch.sortedKeys), func(i int) bool {
        return ch.sortedKeys[i] >= hash
    })
    if idx == len(ch.sortedKeys) {
        idx = 0
    }
    return ch.ring[ch.sortedKeys[idx]]
}
```

 **算法价值** ：节点扩缩容时，仅影响少量数据重分配。

#### 4. 二分查找（Binary Search）—— 日志范围查询

 **业务场景** ：zDBM 日志回放，需找到指定时间点对应的日志位置。

**go**

```
// 二分查找日志位置
func FindLogPosition(logs []*LogEntry, targetTime time.Time) int {
    left, right := 0, len(logs)-1
    for left <= right {
        mid := (left + right) / 2
        if logs[mid].Timestamp.Before(targetTime) {
            left = mid + 1
        } else if logs[mid].Timestamp.After(targetTime) {
            right = mid - 1
        } else {
            return mid
        }
    }
    return right  // 返回最接近的位置
}
```

 **算法价值** ：O(log n) 查找，快速定位恢复点。

#### 5. 排序算法（Sort）—— 任务优先级排序

 **业务场景** ：任务调度器，按优先级和创建时间排序任务。

**go**

```
// 多关键字排序
type Task struct {
    Priority  int
    CreatedAt time.Time
    ID        string
}

// 按优先级（高→低）、创建时间（早→晚）排序
func SortTasks(tasks []Task) {
    sort.Slice(tasks, func(i, j int) bool {
        if tasks[i].Priority != tasks[j].Priority {
            return tasks[i].Priority > tasks[j].Priority
        }
        if tasks[i].CreatedAt != tasks[j].CreatedAt {
            return tasks[i].CreatedAt.Before(tasks[j].CreatedAt)
        }
        return tasks[i].ID < tasks[j].ID
    })
}
```

 **算法价值** ：O(n log n) 排序，支撑任务队列优先级管理。
